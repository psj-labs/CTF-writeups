# Playing With Login - Writeup

## 1. Overview

- 이 문제는 로그인 및 비밀번호 재설정 기능이 존재하는 웹 애플리케이션에서  
  Unicode 처리 방식 차이와 서비스 버전 혼재로 인해 발생하는 인증 우회 취약점을 이용하는 문제입니다.
- 목표는 admin 계정을 탈취하여 flag를 획득하는 것입니다.

- Guest로 로그인한 메인 화면입니다.

![14](/Web/imgs01/14.png)

- 소스코드를 확인하면 init.sql 파일에서 기본 계정이 생성되는 것을 확인할 수 있습니다.
```
    INSERT INTO users (username, password_hash) VALUES ("admin", "...");
    INSERT INTO users (username, password_hash) VALUES ("guest", "...");
```
- 비밀번호는 해시화되어 저장되어 있으며  
  서비스에는 다음 기능이 존재합니다.

- 1. Login
- 2. Password Reset

#### 따라서 admin 계정을 탈취하는 것이 목표임을 알 수 있습니다.

- v1은 Python dictionary 기반 인증 시스템입니다.
```

```
- 즉 메모리에 저장된 users 딕셔너리를 이용해 인증을 수행합니다.

## v2 Login

# 5. Password Reset Flow

- 비밀번호 재설정 과정은 다음과 같습니다.

2. 서버에서 랜덤 token 생성

# 7. Vulnerability Analysis

- 이 문제의 핵심 취약점은 **MySQL Collation과 Unicode 처리 차이**입니다.

- MySQL DB 초기화 과정에서 collation 설정이 존재하지 않기 때문에  
특정 Unicode 문자가 ASCII 문자와 동일하게 비교됩니다.

### 예시
```
    admin
    admín
```
- 위 두 문자열은 MySQL에서는 동일하게 비교될 수 있습니다.

즉
```
    í → i
```
로 처리되는 상황이 발생합니다.

- 하지만 서비스 버전에 따라 문자열 비교 방식이 다릅니다.
```
v1

    Python dictionary 문자열 비교
```
```
v2

    MySQL collation 기반 비교
```
- 따라서 다음과 같은 상황이 발생합니다.
```
    admín == admin
    (MySQL 기준)
```

# 8. Attack Strategy

- 공격 전략은 다음과 같습니다.

1. v1에서 Unicode username 계정 생성
2. v2에서 password reset 요청
3. MySQL collation 때문에 admin 계정으로 인식
4. reset link가 v1 inbox로 전달
5. admin password 변경

## 4. Password Change

- reset 링크는 다음코드로 인해 POST 요청으로 동작합니다.
```python
@app.route("/v2/change-password/<token>", methods=["POST"])
def password_reset_v2(token):
    try:
        row = db_get_valid_token(token)
    except Exception as e:
        print(f"[ERROR] DB token read failed: {e}")
        row = None

    if not row:
        abort(501)

    username = row["username"]
    if request.method == "POST":
        new_pw = request.form.get("new_password", "")
        confirm_pw = request.form.get("confirm_password", "")
        if not new_pw or not confirm_pw:
            abort(501)
        if new_pw != confirm_pw:
            abort(501)

        db_update_user_password(username, new_pw)
        try:
            db_mark_token_used(row["id"])
        except Exception as e:
            print(f"[WARN] Could not mark token used: {e}")

    abort(501)
```


- 따라서 BurpSuite에서 요청 메소드를  
GET → POST 로 변경합니다.

![26](/Web/imgs01/26.png)

- 정상적으로 처리되면 `app.py`로직으로 인해 501 Not Implemented 페이지가 표시되지만  
내부적으로는 비밀번호 변경이 완료됩니다.

# 11. Root Cause

- 이 취약점은 다음 세 가지 문제가 결합되어 발생합니다.

1. MySQL Collation 문제  
   Unicode 문자가 ASCII 문자와 동일하게 비교됨

2. v1 / v2 서비스 로직 혼재  
   서로 다른 인증 시스템 사용

3. 공유된 inbox 구조  
   토큰 전달 경로가 공유됨

---

# 12. Mitigation

- Unicode Normalization을 적용하여 문제를 해결할 수 있습니다.
```
    import unicodedata
    username = unicodedata.normalize("NFKC", request.form.get("username", "")).strip()
```
### 또는

- DB collation 명확하게 설정
- username validation 적용
- 서비스 버전 분리

등을 통해 취약점을 방지할 수 있습니다.
